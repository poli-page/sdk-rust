## 0. TL;DR

We are shipping the Rust sibling of `@poli-page/sdk`. The contract is fixed (`sdk-specification.md` v1.3); we are translating an already-shipped TypeScript reference implementation into idiomatic Rust. The crates.io package is `poli-page`; the library crate name (Rust identifier) is `poli_page` (hyphens are illegal in Rust identifiers, same Python-style split).

The plan: single `cdylib`-less library crate built with stable Cargo. **Async-first** built on `reqwest` (the de-facto Rust HTTP client, used by anthropic-rust, openai-rust, stripe-rust), with an opt-in `blocking` Cargo feature that exposes a sync `PoliPageBlocking` wrapper (mirrors reqwest's own feature design). **Native `enum` for project-mode vs inline sum types** — Rust has real sum types; this is the only port that doesn't need a workaround. `serde` + `serde_json` with `#[serde(rename_all = "camelCase")]` for wire translation (zero hand-written translators — the derive macros do it). `thiserror`-derived `Error` enum with structured variants — single error type with variants for `Auth`, `RateLimited`, `BadRequest`, `NotFound`, `Gone`, `Connection`, `Timeout`, `Aborted`, `Internal`. `tracing` for observability (Rust-ecosystem standard, the equivalent of `log/slog` for Go and PSR-3 for PHP). `wiremock` for unit tests; integration tests against `api-develop.poli.page` gated by env var. CI on GitHub Actions across **Rust stable + beta** on ubuntu, macOS, windows, with MSRV pinned at **1.82** (the floor matches what `reqwest`, `serde`, and `tokio` currently support).

Manual local pre-flight via `scripts/release.sh` is the **primary publishing path** (per engineering guide §6.1). `cargo publish` runs from the maintainer's machine using a crates.io API token stored locally (`~/.cargo/credentials.toml`) — there is no Trusted-Publishing OIDC equivalent for crates.io yet, and `cargo publish` is intentionally not exposed to CI (matches the Go and PHP plans' posture: no auto-publish workflow). After publish, `docs.rs` auto-builds and hosts the rustdoc API reference (the Rust equivalent of pkg.go.dev — zero config).

Ship in **7 phases** (matches Go and PHP — no separate async phase because the async client IS the primary surface; the blocking wrapper is generated by `block_on`). Target: feature-parity 1.0.0 release, behavior-identical to `@poli-page/sdk@1.0.0`. **"Behavior parity" specifically means: same retry policy (5xx+429+network+timeout, jitter `[0.5,1.5)`, Retry-After cap 30s), same error codes round-tripped verbatim, same `is_auth_error()` covering 401+403, same constructor validation, same hooks-never-break-the-request, same project-mode-only constraint on `render.pdf`/`render.pdf_stream`/`render.document`, same primitive-only `RenderMetadata`, same thumbnails wire wrap/unwrap, same `documents.preview` text/html + `X-Document-Page-Count` parsing.**

| **Client type** | `poli_page::PoliPage` (async, primary). `poli_page::blocking::PoliPage` behind the `blocking` feature. |

Reasoning: the **transport core** (URL/header building, response parsing, retry math, error classification) lives in `src/internal/http.rs` as `pub(crate)` pure functions. `pub(crate)` is Rust's real internal-visibility primitive — packages outside the crate cannot import it (unlike Python's `_module` convention or PHP's `Internal\` namespace, which are conventions only).

The async `PoliPage` client in `src/client.rs` orchestrates retries and calls the transport core; namespace types in `src/render.rs` and `src/documents.rs` hold an `Arc<ClientInner>` and don't know about HTTP details. The `blocking` module is a thin wrapper that wraps every async method in `tokio::runtime::Runtime::block_on`.

This mirrors Node's `internal/http.ts` (pure) + `index.ts` (orchestration) + `render.ts`/`documents.ts` (namespace impls) split. Same architecture, different idioms.

### 3.2 The transport seam

```rust
// src/internal/http.rs — pub(crate) only

#[derive(Debug)]
pub(crate) struct HttpAttempt<'a> {
    pub method: http::Method,
    pub path: &'a str,
    pub body: Option<&'a serde_json::Value>,
    pub idempotency_key: Option<&'a str>,
    pub timeout: std::time::Duration,
}

#[derive(Debug)]
pub(crate) struct ParsedResponse {
    pub status: u16,
    pub headers: http::HeaderMap,
    pub body: bytes::Bytes,
}

// ClientInner holds the reqwest::Client and the resolved ClientConfig.
// Render and Documents namespaces hold `Arc<ClientInner>` and call into it.
pub(crate) struct ClientInner {
    pub config: ClientConfig,
    pub http: reqwest::Client,
}

impl ClientInner {
    pub(crate) async fn execute(&self, attempt: HttpAttempt<'_>) -> Result<ParsedResponse, Error>;
    pub(crate) async fn fetch_bytes(&self, url: &str) -> Result<bytes::Bytes, Error>;
    pub(crate) async fn stream_bytes(&self, url: &str) -> Result<BytesStream, Error>;
}
```

Render and Documents namespaces hold `Arc<ClientInner>` (cheap to clone — the `Arc` increments a refcount; the `reqwest::Client` is itself an `Arc<reqwest::ClientRef>` internally so it shares its connection pool by default).

This is the seam unit tests use: they swap `reqwest::Client` with one pointing at a `wiremock::MockServer` — no separate transport trait needed because `reqwest::Client` already abstracts the wire layer.

**Per the n2 design memory cleaned up in the Node SDK**: design all four verbs (POST/GET/DELETE plus the unauthenticated S3 GET for downloads) from day one. The `execute` method is verb-agnostic via `http::Method`; the wire-level work is the same regardless.

### 3.4 `DocumentDescriptor` and `download_pdf`

Mirrors Node's `attachDownloadPdf` (`render.ts:40-63` + `documents.ts:21-44`). The descriptor exposes a `download_pdf()` method:

```rust
#[derive(Debug, Clone, serde::Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct DocumentDescriptor {
    pub document_id: String,
    pub organization_id: String,
    pub project_id: Option<String>,       // nullable on wire
    pub project_slug: Option<String>,
    pub template_id: Option<String>,
    pub template_slug: Option<String>,
    pub version: Option<String>,
    pub environment: Environment,         // enum: Sandbox | Live
    pub api_key_id: Option<String>,
    pub format: PageFormat,
    pub orientation: Option<Orientation>,
    pub locale: Option<String>,
    pub page_count: u32,
    pub size_bytes: u64,
    pub created_at: String,               // ISO 8601 string; not parsed (no chrono dep)
    #[serde(default)]
    pub metadata: RenderMetadata,         // always non-null; defaults to empty map
    pub presigned_pdf_url: String,
    pub expires_at: String,               // ISO 8601; not parsed

    #[serde(skip)]
    pub(crate) client: Option<Arc<ClientInner>>,  // back-reference; filled after deserialization
}

impl DocumentDescriptor {
    /// Fetch the PDF bytes from `presigned_pdf_url`. The URL has a 15-minute TTL —
    /// if it expired, call `documents.get(&id).await` to refresh and retry.
    ///
    /// # Errors
    /// Returns `Error::Download { .. }` on non-2xx or network failure.
    pub async fn download_pdf(&self) -> Result<bytes::Bytes, Error> {
        let client = self.client.as_ref().ok_or(Error::InvalidOptions {
            message: "DocumentDescriptor missing client back-reference".into(),
        })?;
        client.fetch_bytes(&self.presigned_pdf_url).await
    }
}
```

**Nullability**: every wire field that is `string | null` on the API is `Option<String>` in Rust. `None` ↔ JSON `null` (via `serde`'s default `Option` handling, no `omitempty` equivalent needed).

**Metadata is always non-null**: a `RenderMetadata` (which wraps `HashMap<String, MetadataValue>` — see §5.4) with `#[serde(default)]` so missing-on-wire deserializes as empty (the server always echoes `{}`, but defense in depth).

**Client back-reference**: `Option<Arc<ClientInner>>` is set by the SDK after `serde` deserializes the wire payload (via a small `attach_client` helper called inside `render.document` and every `documents.*` method). `#[serde(skip)]` means it's not part of the wire shape — `serde` ignores it on serialize and uses `Default` on deserialize. `download_pdf()` borrows the parent's `reqwest::Client` (so injected TLS / middleware all apply) — but the request to the S3 URL is **unauthenticated** and **not subject to the SDK's retry policy**.

The `blocking::DocumentDescriptor` is the same struct with a `download_pdf` method that internally calls `block_on(self.async_descriptor.download_pdf())`. In practice we expose ONE `DocumentDescriptor` type and gate the `download_pdf` impl on async-vs-blocking via separate `impl` blocks; the blocking variant lives in `src/blocking/mod.rs`.

## 4. Concurrency model

**Async-first with an opt-in `blocking` feature.** This mirrors `reqwest`'s own design, used by every major Rust SDK (anthropic-rust, openai-rust, stripe-rust, aws-sdk-rust).

- **`poli_page::PoliPage`** — async, requires a tokio runtime in scope. Default for FastAPI-equivalent async servers (axum/actix), CLI tools using `#[tokio::main]`, long-running daemons.
- **`poli_page::blocking::PoliPage`** (gated by `features = ["blocking"]`) — synchronous wrapper. Internally constructs a single-threaded tokio runtime owned by the client and calls `block_on` for every method. For scripts, sync-only consumers, or callers who don't want to opt into the async ecosystem.

**Do not** auto-detect "am I in an async context" and switch — that's the requests/trio anti-pattern. Force the user to pick at the import/feature-flag level.

Both clients share the same constructor options (`api_key`, `base_url`, `max_retries`, `retry_delay`, `timeout`, hooks). Both expose the same `render` and `documents` namespaces with identical method signatures (modulo `async` / `.await`).

**Concurrent rendering** is free in async — `tokio::join!` or `futures::future::join_all` over a `Vec<Future>` works out of the box because `Arc<ClientInner>` is `Send + Sync` and `reqwest::Client` pools connections automatically. The blocking wrapper is best used in single-threaded contexts; users wanting concurrent blocking calls can use `std::thread::spawn` plus `Clone`d clients.

**`PoliPage` is `Clone`** — cloning is cheap (`Arc::clone` on the inner state). Recommended pattern: build one client at startup, clone it freely into request handlers / spawned tasks. Document this prominently in the README.

## 5. HTTP transport (reqwest)

### 5.1 Dependency choice

**reqwest is the only runtime HTTP dep**. Reasoning: it's the de facto Rust HTTP client, used by every major Rust SDK and most production Rust services. Async-native, sync wrapper available via its own `blocking` module, mature timeouts, configurable TLS backend, supports streaming response bodies, and integrates with the tokio ecosystem natively.

Minimum supported reqwest: `>=0.12,<0.13` (the 0.12.x line is current at plan time; pin upper bound — reqwest does not guarantee semver stability before 1.0). Mirror anthropic-rust's pin posture.

`Cargo.toml` runtime deps:

```toml

**TLS backend choice — `rustls` by default** (with a `native-tls` escape hatch). Reasoning: rustls is pure-Rust, no system-OpenSSL dependency, cross-compiles cleanly, and matches anthropic-rust / openai-rust defaults. Users on platforms where the system trust store matters (e.g., corporate proxies with MITM certificates managed by IT) can opt into `native-tls` via the feature flag.

**Zero unsafe code** — assert in `lib.rs` with `#![forbid(unsafe_code)]`. Catches accidental `unsafe` blocks at compile time; signals our supply-chain posture.

### 5.2 Connection pooling

Delegated to `reqwest::Client`. The client is internally an `Arc<ClientRef>` that holds the connection pool; cloning the `reqwest::Client` (or our `PoliPage` which wraps it in an `Arc<ClientInner>`) shares the pool. One `PoliPage` per process is the recommended pattern — clone it freely into handlers and tasks. Document this in the README.

The SDK does NOT expose `WithHttpClient`-style injection at v1.0. Adding a `with_http_client(client: reqwest::Client)` builder method later is non-breaking and can land as a 1.x minor if users need it (e.g., for shared connection pools across multiple SDKs). The trade-off is that exposing `reqwest::Client` in the public API pins us to the reqwest major; preserving the option for the future.

### 5.3 Headers (spec §9.1)

Every SDK-originated request includes:

| Header | Value |
|---|---|
| `Authorization` | `Bearer <api_key>` |
| `Content-Type` | `application/json` (POST only — omit on GET/DELETE) |
| `Accept` | `application/json` (always — Node retired `application/pdf` direct responses; see `src/internal/http.ts:79`) |
| `User-Agent` | `poli-page-sdk-rust/<version>` (version from `env!("CARGO_PKG_VERSION")`) |
| `Idempotency-Key` | UUID4 (POST only) — auto-generated unless caller supplies one via `ProjectModeInput::idempotency_key` |

Built by `internal::http::build_headers(method, api_key, idempotency_key, user_agent) -> http::HeaderMap`. Tests verify the matrix per method.

### 5.4 Wire body translation

`serde`'s `#[derive(Serialize, Deserialize)]` + `#[serde(rename_all = "camelCase")]` is the entire wire-translation layer. No hand-written `to_wire`/`from_wire`.

```rust
#[derive(Debug, Clone, Default, serde::Serialize)]
#[serde(rename_all = "camelCase")]
pub struct ProjectModeInput {
    pub project: String,
    pub template: String,
    pub data: serde_json::Value,

    #[serde(skip_serializing_if = "Option::is_none")]
    pub version: Option<String>,

`#[serde(skip)]` is the Rust equivalent of Python's `_to_wire` strip step. `idempotency_key` and `timeout` are read by the SDK and never serialized into the body.

**Thumbnails body wrapping** (deployed API quirk): `POST /v1/documents/:id/thumbnails` wants `{"thumbnails": {...options}}`. The SDK wraps in the `Documents::thumbnails` method body (matches Node `documents.ts:96`):

```rust
let wire = serde_json::json!({ "thumbnails": &options });
```

**Thumbnails response unwrapping**: server returns `{"thumbnails": [...]}`; the SDK deserializes into `struct ThumbnailsResponse { thumbnails: Vec<Thumbnail> }` and returns just the `Vec` (mirror Node `documents.ts:99`).

**Document ID path interpolation** MUST be URL-encoded via `urlencoding::encode(&id)` — or use the `url` crate's `Url::path_segments_mut().push(id)` which percent-encodes per-segment. Port the `encodeURIComponent` calls from Node `documents.ts`.

**`RenderMetadata` constraint**: values are primitives only. Modeled as:

```rust
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]

pub enum MetadataValue {
    String(String),
    Int(i64),
    Float(f64),
    Bool(bool),

pub type RenderMetadata = std::collections::HashMap<String, MetadataValue>;

Mirrors Node's `types.ts:31` constraint and the test at `types.test-d.ts:62-68`.

### 5.5 Response handling

- All SDK endpoints return JSON, except `documents.preview` which returns `text/html`.
- 2xx JSON: `serde_json::from_slice::<T>(body)`. Errors during deserialization become `Error::Internal { code: "INTERNAL_ERROR", ... }`.
- 2xx text/html (`documents.preview` only): `String::from_utf8(body.to_vec())?`; parse `X-Document-Page-Count` via `headers.get("X-Document-Page-Count").and_then(|v| v.to_str().ok()).and_then(|s| s.parse::<u32>().ok()).unwrap_or(0)`. On missing or unparseable header, `page_count = 0` (NaN-tolerant, port `documents.ts:75-77`).
- non-2xx: parse body via `internal::http::parse_error_body(body, status)`, return `Error::<variant>`.
- The presigned-PDF fetch (`download_pdf`, `pdf_stream` second hop) is a plain `reqwest::Client::get(presigned_url).send().await` — **not authenticated, not subject to the retry policy**. Failures map to `Error::Download { status, source }`.

### 5.6 Streaming

`render.pdf_stream` returns a stream of `Bytes` chunks. The trait choice:

```rust
use bytes::Bytes;
use futures_core::Stream;

impl Render {
    pub async fn pdf_stream(
        &self,
        input: ProjectModeInput,
    ) -> Result<impl Stream<Item = Result<Bytes, Error>> + Send + 'static, Error>;
}
```

This is what `reqwest::Response::bytes_stream()` returns; we wrap it with a `map_err` to translate reqwest errors into our `Error::Download` variant. Composes with `tokio::io::copy_buf` for file writes, `futures::pin_mut` for direct iteration, and any `Stream`-aware sink (S3 uploads, HTTP response forwarding, etc.). For callers that need `AsyncRead` instead (e.g. passing the stream to a writer that takes `AsyncRead` like `tokio::io::copy`), wrap with `tokio_util::io::StreamReader::new(stream)` — zero-copy conversion. Document the pattern in the README rather than re-exporting the wrapper.

Implementation:

1. Call `render.document(input)` to get the descriptor (with `presigned_pdf_url`).
2. `reqwest::Client::get(&descriptor.presigned_pdf_url).send().await`. On error → `Error::Download { code: "DOWNLOAD_FAILED", source: err }`.
3. If response status is non-2xx → `Error::Download { code: "DOWNLOAD_FAILED", status: Some(status), .. }`.
4. If response has no body (`content_length == Some(0)`) → `Error::Internal { code: "INTERNAL_ERROR", .. }` (port the `!response.body` guard from `render.ts:128-130`).
5. Return `response.bytes_stream().map_err(...)`. The stream drops cleanly on caller-side `drop()` — no explicit close needed (this is the Rust-native equivalent of Python's context manager / Node's ReadableStream `.cancel()`).

For the blocking variant: `Render::pdf_stream` returns `Result<impl Read + Send + 'static, Error>` — the `reqwest::blocking::Response` IS already `Read`, so we just return it.

**Documents methods do not accept per-call cancellation** — same posture as Node/Python/Go/PHP. The whole future can be dropped to cancel; `tokio::time::timeout` wraps the call for deadlines.

## 6. Retries

Port `src/index.ts` `#runWithRetry` + `src/internal/http.ts` byte-for-byte:

- Retry on **5xx, 429, network errors, and timeouts**.
- Never retry other 4xx (400, 401, 403, 404, 410, etc.).
- Never retry `Error::Aborted` (from a dropped future / external cancellation token).
- `max_retries` default `2` (so up to 3 attempts total). `max_retries: 0` disables retries.
- `retry_delay` default `Duration::from_millis(500)`. `Duration` is type-safe (no millis-vs-seconds confusion), but we surface it in the docs as "500 ms" for consistency with Node.
- Exponential backoff: `retry_delay * 2^(attempt-1)`.
- **Jitter**: multiply by uniform random factor in `[0.5, 1.5)`. Implementation: `let factor = 0.5 + rand::random::<f64>(); let delay = base.mul_f64(2f64.powi(attempt as i32 - 1) * factor);`. NOT ±20%. Use `rand::random` which uses `ThreadRng` seeded from the OS — fast and contention-free across threads.
- **`Retry-After`**: when the response includes the header, use that value as-is (no jitter). Accepts integer seconds or HTTP-date. For HTTP-date use `httpdate::parse_http_date(value).ok()` (the `httpdate` crate is 100 LoC, zero deps — add as a hard dep, ~10 KB). **Capped at 30 seconds** (`const RETRY_AFTER_CAP: Duration = Duration::from_secs(30)`). Past dates clamp to `Duration::ZERO`. Unparseable values are ignored (fall back to computed backoff).
- **The retry sleep MUST be cancellable mid-flight.** Rust-native: `tokio::time::sleep(delay).await` is automatically cancellable — dropping the outer future cancels it. No extra plumbing needed (this is one of the few places Rust is cleaner than Go's `select { case <-time.After(d): case <-ctx.Done(): }`).

When retries are exhausted, fire `on_error` hook with the last `Error` and return it.

Add `httpdate` to the runtime deps:

```toml
httpdate = "1.0"
rand     = "0.8"  # for ThreadRng jitter
```

Or, to keep the dep count minimal, hand-roll a small Retry-After date parser (~20 lines) and use `getrandom` directly instead of pulling `rand`. **Decision**: use `httpdate` (it's a tiny, single-purpose, zero-dep crate maintained by `hyperium` — the same org behind `hyper` / `http` / `reqwest`, so it's already in the transitive graph) and `rand` (already a transitive dep of `tokio` / `reqwest` via `rand_core`).

## 7. Errors

**Single `Error` enum with `thiserror` derive.** Rust's enums are real sum types, so we get the full hierarchy in one type without the inheritance gymnastics Python/PHP/Java need. Pattern-matching is the idiomatic dispatch mechanism; the predicate methods are kept for spec parity.

```rust
#[derive(Debug, thiserror::Error)]
pub enum Error {
    // --- Reserved (SDK-internal) variants — status is None ---

    #[error("invalid options: {message}")]
    InvalidOptions { message: String },

    #[error("network error: {message}")]
    Connection {
        message: String,
        #[source]
        source: Box<dyn std::error::Error + Send + Sync>,
    },

    #[error("request timed out after {timeout:?}")]
    Timeout { timeout: std::time::Duration },

    #[error("request was aborted")]
    Aborted,

    #[error("download failed: {message}")]
    Download {
        message: String,
        status: Option<u16>,
        #[source]
        source: Option<Box<dyn std::error::Error + Send + Sync>>,
    },

    #[error("internal error: {message}")]
    Internal {
        message: String,
        status: Option<u16>,
    },

    // --- API status variants — status is Some(u16), code is the wire `code` field ---

    #[error("bad request ({status}): {message}")]
    BadRequest { status: u16, code: String, message: String, request_id: Option<String> },

    #[error("authentication failed ({status}): {message}")]
    Auth { status: u16, code: String, message: String, request_id: Option<String> },

    #[error("permission denied ({status}): {message}")]
    PermissionDenied { status: u16, code: String, message: String, request_id: Option<String> },

    #[error("not found ({status}): {message}")]
    NotFound { status: u16, code: String, message: String, request_id: Option<String> },

    #[error("gone ({status}): {message}")]
    Gone { status: u16, code: String, message: String, request_id: Option<String> },

    #[error("rate limited ({status}): {message}")]
    RateLimited { status: u16, code: String, message: String, request_id: Option<String> },

    #[error("API error ({status}): {message}")]
    Api { status: u16, code: String, message: String, request_id: Option<String> },  // catch-all for unmatched 4xx/5xx
}
```

The enum is constructed exclusively via `internal::error::classify(status, code, message, request_id)` — a single classifier that maps `(status, code)` pairs to the most specific variant. Tests assert the matrix.

### 7.1 Predicate helpers (kept for spec parity)

Spec §7.1 mandates these predicates. The Rust-idiomatic path is `matches!(err, Error::Auth { .. })` or `if let Error::Gone { .. } = err`; the predicates are convenience methods documented as such:

```rust
impl Error {
    pub fn is_auth_error(&self) -> bool {
        matches!(self, Error::Auth { .. } | Error::PermissionDenied { .. })  // 401 + 403, per spec
    }
    pub fn is_rate_limit_error(&self) -> bool { matches!(self, Error::RateLimited { .. }) }
    pub fn is_validation_error(&self) -> bool { matches!(self, Error::BadRequest { .. }) }
    pub fn is_network_error(&self) -> bool {
        matches!(self, Error::Connection { .. } | Error::Timeout { .. })
    }
    pub fn is_retryable(&self) -> bool {
        if matches!(self, Error::Aborted) { return false; }
        if self.is_network_error() { return true; }
        if let Some(s) = self.status() { return s >= 500 || s == 429; }
        false
    }

    pub fn status(&self) -> Option<u16> {
        match self {
            Error::BadRequest { status, .. }
            | Error::Auth { status, .. }
            | Error::PermissionDenied { status, .. }
            | Error::NotFound { status, .. }
            | Error::Gone { status, .. }
            | Error::RateLimited { status, .. }
            | Error::Api { status, .. } => Some(*status),
            Error::Internal { status, .. } => *status,
            Error::Download { status, .. } => *status,
            _ => None,
        }
    }

    pub fn code(&self) -> &str {
        match self {
            Error::InvalidOptions { .. } => "invalid_options",
            Error::Connection { .. }     => "network_error",
            Error::Timeout { .. }        => "timeout",
            Error::Aborted               => "aborted",
            Error::Download { .. }       => "DOWNLOAD_FAILED",
            Error::Internal { .. }       => "INTERNAL_ERROR",
            Error::BadRequest { code, .. }
            | Error::Auth { code, .. }
            | Error::PermissionDenied { code, .. }
            | Error::NotFound { code, .. }
            | Error::Gone { code, .. }
            | Error::RateLimited { code, .. }
            | Error::Api { code, .. } => code,
        }
    }

    pub fn request_id(&self) -> Option<&str> {
        match self {
            Error::BadRequest { request_id, .. }
            | Error::Auth { request_id, .. }
            | Error::PermissionDenied { request_id, .. }
            | Error::NotFound { request_id, .. }
            | Error::Gone { request_id, .. }
            | Error::RateLimited { request_id, .. }
            | Error::Api { request_id, .. } => request_id.as_deref(),
            _ => None,
        }
    }
}
```

### 7.3 Error-body parsing (port `parseErrorBody` exactly)

```rust
pub(crate) fn parse_error_body(body: &[u8], status: u16) -> (String, String) {
    #[derive(serde::Deserialize)]
    struct ErrorBody {
        code: Option<String>,
        message: Option<String>,
        error: Option<String>,
    }

    let parsed: Result<ErrorBody, _> = serde_json::from_slice(body);
    match parsed {
        Ok(b) => {
            let code = b.code
                .or(b.message.clone())
                .or(b.error)
                .unwrap_or_else(|| "unknown_error".to_string());
            let message = b.message
                .unwrap_or_else(|| format!("API error ({status}): {code}"));
            (code, message)
        }
        Err(_) => (
            "INTERNAL_ERROR".to_string(),
            format!("API error {status}: response body was not valid JSON"),
        ),
    }
}

## 8. Cancellation and timeouts

Rust has two complementary cancellation mechanisms; the SDK supports both:

- **Future cancellation**: dropping the future cancels the in-flight request (reqwest cooperates with tokio's cancellation). This is the zero-config path — `tokio::select!` / `tokio::time::timeout` / aborting a `JoinHandle` all work naturally.
- **Per-request timeout**: `timeout` constructor option default `Duration::from_secs(60)` — applied to the whole attempt (network + parse). Per-call override via `ProjectModeInput::timeout`. Internally we use `tokio::time::timeout(timeout, fut).await`.
- **Timeout scope is per-attempt, not per-call**. A call with `max_retries: 2`, `timeout: 60s`, and a server returning Retry-After can take up to `3 × 60s + 2 × RETRY_AFTER_CAP = 240s` wall-clock worst case. Document this prominently in the README's "Retries & timeouts" section — silent "60s timeout took 4 minutes" is the kind of surprise that ends an SDK's adoption.
- **Overall elapsed budget** (`max_elapsed_total: Option<Duration>` on the builder, default `None`): when set, the retry loop aborts with `Error::Timeout { kind: TimeoutKind::Overall, .. }` once cumulative elapsed time exceeds the budget — including mid-`Retry-After`-sleep (the sleep is `tokio::select!`'d against a deadline). Pattern borrowed from `aws-sdk-rust`'s `RetryConfig::max_elapsed_time`. Use this when the caller has a hard SLA (e.g., serving an inbound request with its own deadline).

```rust
let pdf = tokio::time::timeout(
    Duration::from_secs(10),
    client.render.pdf(input),
).await
    .map_err(|_| Error::Timeout { timeout: Duration::from_secs(10) })??;
```

**Error mapping** (port the abort-vs-timeout distinction from Node `index.ts:209-218`):

- `reqwest::Error::is_timeout()` → `Error::Timeout { timeout: ... }`. Retryable.
- `reqwest::Error::is_connect()` / `is_request()` / network-level failures → `Error::Connection { source, .. }`. Retryable.
- The future is dropped (e.g., outer `tokio::select!` chose another branch) → no error is returned to the caller because no one is awaiting; if a cancellation token is involved and explicitly checked, raise `Error::Aborted`. Never retryable.
- `tokio::time::error::Elapsed` from the SDK's own per-attempt timeout → `Error::Timeout { timeout: ... }`. Retryable.

**Documents methods do not accept per-call cancellation overrides.** They use the client-level `timeout` as passed. Mirrors Node, which only puts `signal` on `BaseRenderInput`.

**Mid-sleep cancellation is free**: `tokio::time::sleep(delay).await` is cancellable by dropping the surrounding future. Port Node's `#sleep` behavior at `index.ts:245-260` — Rust's async runtime gives us this without any extra plumbing.

## 9. Type system (Rust-specific)

### 9.1 Native sum types for project-mode enforcement

Spec §4.1 requires compile-time enforcement that `render.pdf`/`pdf_stream`/`document` accept project mode only. Rust has native sum types via `enum` — the cleanest expression of this constraint across all the SDKs:

```rust
#[derive(Debug, Clone)]
pub enum RenderInput {
    Project(ProjectModeInput),
    Inline(InlineModeInput),
}

#[derive(Debug, Clone, Default, serde::Serialize)]
#[serde(rename_all = "camelCase")]
pub struct ProjectModeInput { /* ... §5.4 ... */ }

#[derive(Debug, Clone, serde::Serialize)]
#[serde(rename_all = "camelCase")]
pub struct InlineModeInput {
    pub template: String,            // raw HTML
    pub data: serde_json::Value,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub format: Option<PageFormat>,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub orientation: Option<Orientation>,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub locale: Option<String>,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub metadata: Option<RenderMetadata>,
    #[serde(skip)]
    pub idempotency_key: Option<String>,
    #[serde(skip)]
    pub timeout: Option<std::time::Duration>,
}

impl From<ProjectModeInput> for RenderInput {
    fn from(p: ProjectModeInput) -> Self { RenderInput::Project(p) }
}
impl From<InlineModeInput> for RenderInput {
    fn from(i: InlineModeInput) -> Self { RenderInput::Inline(i) }
}

// Method signatures — compile-time enforcement via direct parameter types:
impl Render {
    pub async fn pdf(&self, input: ProjectModeInput)        -> Result<Bytes, Error>;
    pub async fn pdf_stream(&self, input: ProjectModeInput) -> Result<impl Stream<Item = ...>, Error>;
    pub async fn document(&self, input: ProjectModeInput)   -> Result<DocumentDescriptor, Error>;

    // Preview accepts either; use `impl Into<RenderInput>` for ergonomic call sites.
    pub async fn preview(&self, input: impl Into<RenderInput>) -> Result<PreviewResult, Error>;
}
```

`client.render.pdf(InlineModeInput { .. }).await` is a **compile-time type error** — the parameter type is `ProjectModeInput`, not `RenderInput`. `client.render.preview(project_input)` and `client.render.preview(inline_input)` both compile via the `From` impls.

**Type names mirror Node**: `ProjectModeInput`, `InlineModeInput`, `RenderInput`. Same names as Python (§9.1), Go (§9.1), and PHP (§9.1) so spec users navigating multiple languages find familiar identifiers.

**No runtime defense layer needed** — the type system makes the wrong call impossible to construct. Save the runtime check that Python/Go/PHP need; Rust's enum is closed by construction (no `unsafe` workaround in our public surface — `forbid(unsafe_code)` at the crate root enforces this).

### 9.2 Nullable wire fields → `Option<T>`

Every JSON `string | null` field maps to `Option<String>` in Rust. `serde` handles `None` ↔ `null` translation natively. `#[serde(skip_serializing_if = "Option::is_none")]` on outgoing fields prevents `"field": null` from leaking onto the wire when the value is `None` (saves bytes, matches Node's `omitempty` Go equivalent).

No `Ptr<T>` / `Opt[T]` helper needed (unlike Go) — `Option<T>` is the language primitive.

**Forward-compat posture on response types**:

- **Never** apply `#[serde(deny_unknown_fields)]` to response structs. The server adding a field should be a silent no-op for old SDK versions, not a deserialization error. `stripe-rust`, `anthropic-sdk-rust`, and `aws-sdk-rust` all follow this rule explicitly. (Request types are unaffected — we don't currently use `deny_unknown_fields` there either.)
- **Enums the server may extend** (`PageFormat`, `Orientation`, any future status/kind enums) MUST include a catch-all variant so a server-side addition doesn't hard-fail every deployed SDK. `#[serde(other)]` is the simplest path:

  ```rust
  #[derive(Debug, Clone, serde::Deserialize, serde::Serialize)]
  #[serde(rename_all = "UPPERCASE")]
  pub enum PageFormat {
      A4, A3, Letter, Legal,
      /// Catch unknown values so the SDK doesn't break when the API adds a new format.
      #[serde(other)]
      Unknown,
  }
  ```

  Caveat: `#[serde(other)]` only works on unit variants and discards the raw value. To preserve the unknown string for debugging, use a tuple variant + a hand-written `Deserialize` impl (or `serde_with::PickFirst`). For v1.0, the unit-variant form is sufficient — debugging an unknown enum value is a server-side investigation anyway.

### 9.3 Linters and static checks

- **`cargo fmt --check`** — rustfmt, enforced in CI and pre-push.
- **`cargo clippy --all-targets --all-features -- -D warnings`** — clippy is the Rust ecosystem's industry-standard linter. Mandatory in CI. Selectively enable pedantic lints in `clippy.toml`.
- **`cargo doc --no-deps --all-features -D warnings`** — `-D warnings` makes broken intra-doc links a CI failure. Mirrors the doc-comment-required policy on Go's `revive` and PHP's PHPStan-with-`@throws`.
- **`cargo deny check`** — license policy (only MIT / Apache-2.0 / BSD permissive), advisory check (RustSec database), source registry check (only crates.io).
- **`cargo audit`** — RustSec advisory check on `Cargo.lock`. Redundant with `cargo deny check advisories` but slightly faster; we run `cargo deny` in CI and ship `cargo audit` as a dev tool.
- **`cargo public-api --diff`** — detects breaking changes to the public API surface across versions. CI compares the current branch's public surface against the latest published crate; mismatch on a non-major-bump PR fails CI. This is the Rust equivalent of TypeScript's `@arethetypeswrong/cli` for npm packages.

`#![forbid(unsafe_code)]` at the crate root. `#![warn(missing_docs)]` at the crate root (and `-D warnings` in CI) enforces a doc comment on every public symbol — Rust's equivalent of Go's `revive` enforcement.

### 9.4 Rustdoc requirements

Rust's documentation system is non-negotiable for SDKs — `docs.rs` IS the reference.

- **Every public `type`, `fn`, `method`, `const`, `mod` must have a `///` doc comment.**
- Doc comments start with a one-line summary, blank line, then longer description.
- Code examples in doc comments are compiled as `doctests` (`cargo test --doc`). Every public method's doc comment includes a runnable example.
- `#![warn(missing_docs)]` + `-D warnings` in CI enforces presence; rustdoc itself enforces correctness via doctests.
- `docs.rs` automatically builds and hosts the rustdoc HTML for every published version — zero config beyond `[package.metadata.docs.rs]` in `Cargo.toml` to enable all features for the build (so `blocking` examples render too).

```toml
[package.metadata.docs.rs]
all-features = true
rustdoc-args = ["--cfg", "docsrs"]
```

## 10. Observability

Modern Rust SDKs (anthropic-rust, openai-rust, aws-sdk-rust) ship **`tracing` integration as the primary observability mechanism**, with SDK-level hooks reserved for retry/error decisions. We follow the same pattern.

### 10.1 `tracing` integration

```rust
use tracing::{debug, info, warn, error};

// Every HTTP attempt emits a tracing span:
let span = tracing::debug_span!("polipage.request",
    method = %req.method(),
    url    = %req.url(),
    attempt = attempt,
    request_id = tracing::field::Empty,  // filled in once we have a response (success or error)
);
let _enter = span.enter();
debug!("sending request");
// ... do the request ...
if let Some(rid) = resp.headers().get("x-request-id").and_then(|v| v.to_str().ok()) {
    span.record("request_id", rid);       // visible on both success and error spans
}
debug!(status = resp.status().as_u16(), "received response");
```

- **Silent by default**: `tracing` events emit nothing unless the user installs a subscriber. Adding a subscriber is one line for users using `tracing-subscriber`:
  ```rust
  tracing_subscriber::fmt::init();
  ```
- **One DEBUG event per HTTP attempt** (method, url, status, duration_ms, attempt). One INFO event per response. One WARN event per retry. One ERROR event per terminal failure.
- **Redaction**: never include the `Authorization` header value in any field, never include the api_key in any structured field. Apply a field-name allowlist when serializing request body for the `tracing::debug!` event — we only emit `method`, `url`, `status`, `duration_ms`, `attempt`, `request_id`. Body content is not logged.
- **Request ID on success spans**: when the response includes `x-request-id`, record it on the span via `span.record("request_id", value)` (see code block above). Pattern from `cloudflare-rs` and `aws-sdk-rust` — log correlation works on the 99% happy path, not just on errors.
- Works seamlessly with `tracing-subscriber` (fmt, json), `tracing-opentelemetry` (OTLP export), `tracing-bunyan-formatter` (structured logs), etc. The ecosystem already has plug-ins for every observability backend.

This is the Rust equivalent of Python's `logging.getLogger("poli_page")` + `POLI_PAGE_LOG` env var, Go's `log/slog`, and PHP's PSR-3 `LoggerInterface` injection.

### 10.2 SDK-level hooks (`on_retry`, `on_error` only)

Set via the builder:

```rust
let client = PoliPage::builder()
    .api_key("pp_test_...")
    .on_retry(|e: &RetryEvent| metrics::inc("polipage.retry"))
    .on_error(|err: &Error| sentry::capture(err))
    .build()?;
```

Hooks are `Arc<dyn Fn(&T) + Send + Sync>` — synchronous callbacks (no `async` in the hook signature; doing async work would require boxed futures and complicate the type signature for marginal benefit). Users wanting async work in a hook can `tokio::spawn(async move { ... })` inside the hook themselves.

**Hooks must never break the request** — every hook invocation is wrapped in `std::panic::catch_unwind`:

```rust
fn fire_hook<T>(hook: &Option<Arc<dyn Fn(&T) + Send + Sync>>, event: &T)
where T: std::panic::RefUnwindSafe
{
    if let Some(h) = hook {
        // Swallow panics — hooks must not break the request. Mirrors Node #fireHook.
        let _ = std::panic::catch_unwind(std::panic::AssertUnwindSafe(|| h(event)));
    }
}
```

Port Node `index.ts:106-113`.

**Timing** (port Node `index.ts` firing points at lines 163, 168):

- `on_retry` fires before each retry sleep, with the delay and the error that triggered the retry.
- `on_error` fires once at terminal failure — retries exhausted, non-retryable error, or aborted.

`on_request` / `on_response` from earlier drafts are **dropped** — users who want request/response inspection install a `tracing` subscriber and consume the spans, OR use reqwest's `Middleware`-style wrapping (via the `reqwest-middleware` crate) in their own code. We don't reinvent that machinery.

**Event types**:

```rust
#[derive(Debug, Clone)]
pub struct RetryEvent {
    pub attempt: u32,                        // 1-based; the attempt about to be made
    pub delay: std::time::Duration,          // sleep duration before this attempt
    pub reason: Error,                       // error that triggered the retry (cloned for the event)
}
```

Note: `Error` derives `Clone` for this purpose (the `Box<dyn Error>` `source` in `Connection` does NOT derive `Clone`, so the cloned event drops the source — but keeps message, code, status; sufficient for observability).

`on_error` receives `&Error` directly (no event wrapper), mirroring Node `onError?: (err: PoliPageError) => void`.

## 11. File helper (spec §2, §5.1 last paragraph)

```rust
/// Render a PDF and stream the bytes to the given path.
/// Creates parent directories if missing. Overwrites existing files.
/// Behavior parity with Node's `renderToFile`.
///
/// # Errors
/// Returns `Error::InvalidOptions` on filesystem failure, or any error from `render.pdf_stream`.
pub async fn render_to_file(
    client: &PoliPage,
    input: ProjectModeInput,
    path: impl AsRef<std::path::Path>,
) -> Result<(), Error> {
    use tokio::io::AsyncWriteExt;
    use futures_util::StreamExt;

    let path = path.as_ref();
    if let Some(parent) = path.parent() {
        tokio::fs::create_dir_all(parent).await.map_err(|e| Error::InvalidOptions {
            message: format!("failed to create directory {}: {}", parent.display(), e),
        })?;
    }

    let mut stream = std::pin::pin!(client.render.pdf_stream(input).await?);
    let mut file = tokio::fs::File::create(path).await.map_err(|e| Error::InvalidOptions {
        message: format!("failed to create file {}: {}", path.display(), e),
    })?;

    while let Some(chunk) = stream.next().await {
        let chunk = chunk?;
        file.write_all(&chunk).await.map_err(|e| Error::InvalidOptions {
            message: format!("failed to write to {}: {}", path.display(), e),
        })?;
    }
    file.flush().await.map_err(|e| Error::InvalidOptions {
        message: format!("failed to flush {}: {}", path.display(), e),
    })?;
    Ok(())
}
```

Free function in the crate root (matches spec §2 — `render_to_file`). Streams chunks via tokio's async I/O; bounded memory regardless of PDF size.

For the blocking variant, mirror in `src/blocking/fs.rs` using `std::fs` and `std::io::copy` against the `impl Read` returned by `blocking::Render::pdf_stream`.

`path` accepts `impl AsRef<Path>` so callers can pass `&str`, `&Path`, `PathBuf`, etc.

**`Cargo.lock` IS committed.** Cargo's own documentation has, since 2023, recommended committing `Cargo.lock` for both binaries AND libraries (the historical "libraries shouldn't commit it" rule was reversed). The reasoning: the lock makes contributors' builds reproducible without affecting downstream consumers (who resolve their own `Cargo.lock` against their constraints). This matches the modern Rust-ecosystem default — reqwest, serde, tokio, hyper, anthropic-rust, and stripe-rust all commit their `Cargo.lock`. We pin a real version not a moving target; bumps to the MSRV are minor-version releases with a clear note in `MIGRATION.md` (engineering guide §3.3).

> **Phase 0 note (MSRV 1.82 → 1.93)**: the spec was authored when stable was ~1.84; the literal `1.82` honored the §12.6 cadence rule (*current stable minus 2 minors*) at that time. By Phase 0 the ecosystem had moved — current stable is 1.95, and contemporary `url`/`idna_adapter`/`icu_*` / `hashbrown 0.17` / `rand 0.10` / `uuid 1.21+` / `zerovec 0.11.6` all require `>= 1.85`. Pinning every offending transitive would have made the lockfile load-bearing and re-broken with every dependabot bump. We followed §12.6's load-bearing rule (current stable minus 2 = 1.93) rather than the literal `1.82` number; the cadence rule is what was always meant to govern.

### 12.4 Distribution

### 12.5 Release process (per engineering guide §6.1)

The **`scripts/release.sh` is the primary publishing path** — a maintainer-driven flow that gates the release on lint/test/confirm-prompt before `cargo publish` + tag push.

1. Bump `version` in `Cargo.toml`.
2. Update `CHANGELOG.md` (move `[Unreleased]` → `[X.Y.Z] - YYYY-MM-DD`).
3. If a MAJOR bump, add a section to `MIGRATION.md`.
4. Commit on `main` (`chore(release): vX.Y.Z`).
5. Run `scripts/release.sh` locally:
   - `cargo fmt --check`
   - `cargo clippy --all-targets --all-features -- -D warnings`
   - `cargo doc --no-deps --all-features` (warnings deny)
   - `cargo test --all-features`
   - `cargo deny check`
   - `cargo audit`
   - Integration tests if `POLI_PAGE_API_KEY` set: `cargo test --features integration --test integration -- --ignored`
   - `cargo run --example demo` end-to-end smoke against develop
   - `cargo publish --dry-run`
   - prompt for confirmation
6. On confirmation: `cargo publish` (reads token from `~/.cargo/credentials.toml`).
7. Wait for the crate to appear on crates.io (~30 seconds).
8. `git tag v1.0.0 && git push origin v1.0.0`. (Tag push is **after** publish, not before — the tag commemorates the published version. Mirrors anthropic-rust / stripe-rust.)
9. `docs.rs` auto-builds within ~5 minutes; visit `https://docs.rs/poli-page/1.0.0` to confirm.
10. Optionally `gh release create v1.0.0 --notes-from-tag` for the GitHub Release page.

**No automatic publish workflow.** crates.io does not yet support OIDC / Trusted Publishing (the `cargo-token-rotation` RFC is in discussion but unmerged at plan time). The maintainer's local `~/.cargo/credentials.toml` is the credential boundary — same posture as Go's manual `git push origin vX.Y.Z` and PHP's manual tag push. The engineering guide §6.4 "no auto-publish on tag push" rule is structurally satisfied because we don't push the tag until after the manual publish succeeds.

### 12.6 Release discipline (per engineering guide §3)

**SemVer 2.0.0** (§3.1). Pre-1.0 versions MAY break in minor bumps. From v1.0 onward: MAJOR for breaking changes to the contract or public API, MINOR for backwards-compatible additions, PATCH for bug fixes. The `cargo public-api --diff` check in CI (§9.3) catches inadvertent breaking changes — a failing diff requires either fixing the change or bumping the major.

**CHANGELOG discipline** (§3.2). Update `CHANGELOG.md` in the **same commit** as the version bump. Use Keep a Changelog sections (`Added`, `Changed`, `Deprecated`, `Removed`, `Fixed`, `Security`). **Mark breaking changes with `**BREAKING**:`** — cross-SDK convention.

**MIGRATION.md authoring** (§3.3). Every MAJOR bump SHOULD add a section to `MIGRATION.md` describing what changed and how to adapt. MSRV bumps within a minor version also get a `MIGRATION.md` note (Rust convention treats MSRV bumps as breaking changes for some maintainers; ours follows the engineering guide's "minor with documented note" model).

**MSRV cadence**. The MSRV pin (`rust-version = "1.82"` in `Cargo.toml`) tracks **current stable minus 2 minor releases** — the same cadence `tokio`, `reqwest`, and `serde` document. With Rust on a 6-week release cycle, that's a ~3-month window. Bumps are MINOR releases (per engineering guide §3) with a `MIGRATION.md` note explaining the new floor. MSRV is enforced two ways: (1) `Cargo.toml`'s `rust-version` field causes `cargo` to refuse to build on older toolchains (Cargo enforces it at resolve time as of 1.74), and (2) the dedicated CI step `cargo +<msrv> build --locked --all-features` (§16.1 step 9). Never bump MSRV opportunistically — only when a transitive dep forces our hand or a language feature delivers concrete value.

**Prerelease channel** (§3.4). Cargo uses semver prerelease suffixes natively (`1.0.0-rc.1`, `2.0.0-beta.0`). `cargo add poli-page` ignores prereleases by default; users opt in by pinning explicitly: `cargo add poli-page@1.0.0-rc.1`. Promotion flow:

1. Bump `version` to the prerelease form (e.g., `2.0.0-rc.1`). Move CHANGELOG entries under that heading.
2. Run `scripts/release.sh`. On confirmation, `cargo publish` uploads it as a prerelease.
3. Tag `v2.0.0-rc.1` and push the tag.
4. When ready, bump `version` to the stable form (drop the suffix), move CHANGELOG entries to the stable heading, run `scripts/release.sh` again, push the stable tag.

Stable and prerelease tags MUST never point at the same commit — once a prerelease is promoted, the next prerelease starts a new pre-suffix sequence.

**License**: **MIT OR Apache-2.0 dual license** (`license = "MIT OR Apache-2.0"` in `Cargo.toml`). This is the prevailing Rust-ecosystem convention — used by every crate in the standard library, every major framework (tokio, serde, axum, actix-web, hyper), and every leading SDK (anthropic-rust, openai-rust, stripe-rust). The dual license maximizes compatibility: users on Apache-restrictive projects pick MIT; users on MIT-restrictive projects pick Apache. Ship both `LICENSE-MIT` and `LICENSE-APACHE` at the repo root; reference the convention in `README.md`'s License section. **This differs from the Node/Python/Go/PHP SDKs** (which ship MIT-only) — but matching the Rust ecosystem norm is more important than cross-SDK license uniformity here, because Rust users routinely check license compatibility tools (`cargo deny`) and dual-licensing is the path of least friction.

## 14. Testing strategy

### 14.1 Unit tests (`tests/` directory — Cargo's integration-test convention)

Rust's `tests/*.rs` files are compiled as separate crates that import the public surface — they're called "integration tests" by Cargo but functionally serve the role of unit tests for the public API. (Rust-internal unit tests live inside `src/` modules under `#[cfg(test)] mod tests`.)

- **`wiremock`** for HTTP mocking — async-native, integrates cleanly with `reqwest`.
- Cover every error code path.
- Test retry loop: backoff math, max attempts, Retry-After honoring (30-second cap, past dates, far-future dates, unparseable values), jitter bounds (`[0.5, 1.5)` factor over 200 samples — port `http.test.ts:83-93`).
- Test wire translation via `serde_json::to_value` / `from_value` round-trip on every input type.
- Test the enum-based `RenderInput`: `client.render.pdf(InlineModeInput { .. })` must fail to compile (verified via `trybuild` — see §14.4).
- Test `parse_error_body` fallback chain (`code → message → error → 'unknown_error'`) including HTML error pages and empty bodies — port `http.test.ts:108-152`.
- Test `is_auth_error()` returns `true` for both 401 (`Error::Auth`) and 403 (`Error::PermissionDenied`).

### 14.2 Integration tests (`tests/integration/`, gated by `--features integration`)

### 14.3 Doctests

Every public method's `///` comment includes a runnable example block. `cargo test --doc` compiles and runs each one — failures fail the build. Example:

````rust
/// Renders a PDF and returns the bytes.
///
/// # Example
///
/// ```no_run
/// # use poli_page::{PoliPage, ProjectModeInput};
/// # #[tokio::main]
/// # async fn main() -> Result<(), Box<dyn std::error::Error>> {
/// let client = PoliPage::new("pp_test_...")?;
/// let pdf = client.render.pdf(ProjectModeInput {
///     project: "billing".into(),
///     template: "invoice".into(),
///     version: Some("1.0.0".into()),
///     data: serde_json::json!({ "invoiceNumber": "INV-001" }),
///     ..Default::default()
/// }).await?;
/// std::fs::write("invoice.pdf", &pdf)?;
/// # Ok(())
/// # }
/// ```
pub async fn pdf(&self, input: ProjectModeInput) -> Result<Bytes, Error> { /* ... */ }
````

`no_run` means the doctest compiles but doesn't execute (it would need a live API). Compile-success is the verification; execution happens in integration tests.

### 14.4 Compile-fail tests via `trybuild`

The Rust equivalent of TypeScript's `tsd` or Python's pyright "must error" harness:

```toml
[dev-dependencies]
trybuild = "1.0"
```

```rust
// tests/compile_fail.rs
#[test]
fn cant_pass_inline_to_pdf() {
    let t = trybuild::TestCases::new();
    t.compile_fail("tests/compile-fail/pdf_with_inline.rs");
    t.compile_fail("tests/compile-fail/pdf_stream_with_inline.rs");
    t.compile_fail("tests/compile-fail/document_with_inline.rs");
    t.compile_fail("tests/compile-fail/metadata_with_nested_object.rs");
}
```

Each file under `tests/compile-fail/` contains the code that MUST fail to compile, plus a `.stderr` file capturing the expected compiler output. `trybuild` runs the compiler in a sandbox and asserts the output matches.

This is defense-in-depth (the type system enforces the constraint at the call site automatically; trybuild verifies it for new contributors who might accidentally widen the parameter type).

## 16. CI/CD

### 16.1 ci.yml (every push + PR)

- Matrix: `{ rust: [stable, beta], os: [ubuntu-latest] }` + one job each on `macos-latest` and `windows-latest` with `stable`. Matrix follows **Rust's actual support model** — the latest stable + one ahead (`beta`) — rather than tracking individual `x.y` numbers. This is the convention used by anthropic-rust, openai-rust, tokio, serde, reqwest. MSRV (1.82) is verified by a dedicated step that uses `cargo +1.82 build` against the pinned `Cargo.lock` (a `rust-version` check, not a separate matrix entry, because Cargo enforces MSRV at resolve time when `rust-version` is set).
- Setup: `dtolnay/rust-toolchain@<sha>` (the de facto action for installing Rust in GH Actions, pin to SHA per §16.6).
- Steps:
  1. `cargo fmt --all --check`
  2. `cargo clippy --all-targets --all-features -- -D warnings`
  3. `cargo doc --no-deps --all-features` (with `RUSTDOCFLAGS=-D warnings` to fail on broken intra-doc links)
  4. `cargo test --all-features` (runs unit + doctest + trybuild)
  5. `cargo test --no-default-features --features blocking` (verify blocking-only build)
  6. `cargo deny check`
  7. `cargo audit`
  8. **Public-API diff** (on PRs only): `cargo public-api --diff "v$(cargo pkgid | sed -nE 's/.*#(.*)/\1/p')" -- --message-format=json | jq` — comparing the branch against the latest published version. Fails the PR if breaking changes are detected without a major-version bump in `Cargo.toml`.
  9. **MSRV check**: `cargo +1.82 build --locked --all-features` — `--locked` is critical: without it, a regenerated `Cargo.lock` can pull dep versions that have silently bumped their own MSRV past 1.82, masking the regression. `cargo-msrv find` is an alternative for exhaustive bisection, but `+1.82 build --locked` is the gate.
  10. **Semver-checks** (on PRs only, complement to step 8): `cargo semver-checks check-release`. Catches semver violations `cargo public-api --diff` misses — non-defaulted trait method additions, lifetime widening, enum variant additions without `#[non_exhaustive]`, etc. Used by tokio, axum, clap. The two tools are complementary; both run.
  11. **Consumer-crate smoke** (mirrors Node CI install-smoke):
      - `mkdir /tmp/smoke && cd /tmp/smoke`
      - `cargo new --lib smoke`
      - Append `poli-page = { path = "$GITHUB_WORKSPACE" }` to `smoke/Cargo.toml`
      - Write a 5-line `lib.rs` that constructs `PoliPage::builder().api_key("test").build().unwrap()`
      - `cargo build`
      - Fails CI if the consumer compile breaks.
- **Integration tests** run in a separate workflow `.github/workflows/integration.yml` (mirrors the Go and PHP siblings' §16.1):
  - **Triggers**: `schedule: cron "0 6 * * *"` (nightly 06:00 UTC, after the daily develop deploy) + `push: branches: [main]` + `workflow_dispatch` (manual run).
  - **Permissions**: `contents: read` only.
  - **Secrets**: `POLI_PAGE_API_KEY` (required), `POLI_PAGE_BASE_URL` (optional, defaults to `https://api-develop.poli.page`).
  - **Env passthrough** (mirror Node integration tests): `POLI_PAGE_TEST_PROJECT` / `POLI_PAGE_TEST_TEMPLATE` / `POLI_PAGE_TEST_VERSION`.
  - **Job**: `cargo test --features integration --test integration -- --ignored --include-ignored` on ubuntu-latest + stable. Single job — integration tests cover wire contract, not platform portability (that's `ci.yml`'s job).
  - **Failure policy**: a single failure opens a GitHub issue via the `peter-evans/create-issue-from-file` action (SHA-pinned per §16.6); doesn't fail subsequent nightlies.
- **npm-script → cargo-tool mapping** (for porters cross-referencing `sdk-node/package.json`):

  | Node script | Rust equivalent |
  |---|---|
  | `lint` | `cargo clippy --all-targets --all-features -- -D warnings` |
  | `format` | `cargo fmt --all` |
  | `typecheck` | `cargo check --all-targets --all-features` (Rust's type system is the build) |
  | `test` | `cargo test --all-features` |
  | `test:types` | `cargo test --test compile_fail` (trybuild harness) |
  | `test:integration` | `cargo test --features integration --test integration -- --ignored` |
  | `build` | `cargo build --release --all-features` |
  | `lint:pack` (attw + publint) | `cargo publish --dry-run` + `cargo deny check` + `cargo public-api --diff` |
  | `size` (size-limit) | n/a — Rust libraries distribute source; binary size depends on consumer flags |
  | `prepare` (husky) | `scripts/install-hooks.sh` writes `.git/hooks/pre-push` |

### 16.2 No release workflow

There is no `release.yml`. crates.io has no Trusted Publishing / OIDC support yet; `cargo publish` requires a long-lived API token. The token lives in the maintainer's `~/.cargo/credentials.toml` and never enters CI. The release flow is `scripts/release.sh` → `cargo publish` → `git tag` → `git push origin <tag>` — all manual, all local. This matches anthropic-rust / openai-rust / stripe-rust.

If crates.io ever ships Trusted Publishing (the [pre-RFC discussion](https://github.com/rust-lang/rfcs/pull/3691) was active at plan time), a future `publish.yml` workflow can be added as the signed-attestation augment (per engineering guide §6.3) — but `scripts/release.sh` stays the primary path.

### 16.3 release.sh (primary publishing path — per engineering guide §6.1)

Mirror `sdk-node/scripts/publish.sh` shape:

1. **Pre-flight**: assert on `main`, working tree clean, target `vX.Y.Z` tag does not yet exist locally or on the remote, `cargo login` token is present.
2. **Verify**: `cargo fmt --check` + `cargo clippy ... -D warnings` + `cargo doc -D warnings` + `cargo test --all-features` + `cargo deny check` + `cargo audit` + integration tests if `POLI_PAGE_API_KEY` set + `cargo run --example demo` end-to-end against develop + `cargo publish --dry-run`.
3. **Confirm**: prompt before publishing. Abort cleanly on `n`.
4. **Publish**: `cargo publish` (no `--dry-run`).
5. **Tag**: `git tag vX.Y.Z && git push origin vX.Y.Z`. (Tag push is after publish, because if publish fails we don't want a tag pointing at an unpublished commit.)
6. **`--dry-run` flag** does everything except `cargo publish` (still runs `cargo publish --dry-run` for safety) and the tag push. CI invokes `--dry-run` on PRs that touch `Cargo.toml` or `Cargo.lock`.

### 16.4 Pre-push hooks (engineering guide §8)

`.git/hooks/pre-push` installed via `scripts/install-hooks.sh` (idempotent). Hooks: `cargo fmt --all --check`, `cargo clippy --all-targets --all-features -- -D warnings`, `cargo test --all-features` (unit + doctest, skips integration). Add a `RUN_INTEGRATION=1` opt-in for full integration runs.

### 16.5 No separate docs workflow

`docs.rs` pulls automatically from published versions. No `docs.yml` needed. If `docs.poli.page/reference/sdk/rust/` is desired for symmetry with Node/Python/PHP, configure a static redirect to `https://docs.rs/poli-page/latest` (Xavier).

### 16.6 Supply-chain protections

Mirror the protections added to the Node SDK in commit `bd85a76` ("Adds safety measures against supply chain attacks") and the sibling plans' §16.6. The Rust ecosystem provides several protections natively; we layer the rest.

**Publish-side**:
- **Manual `cargo publish` after local `scripts/release.sh` confirmation**. No CI token, no automatic publish workflow. The token lives in `~/.cargo/credentials.toml` on the maintainer's machine, scoped to the `poli-page` crate via crates.io's per-crate token feature (set up once via the crates.io UI: "API Tokens → New Token → Scope: poli-page only").
- **Crates.io immutability**: published versions are immutable. Yanking marks a version as broken but doesn't remove it (source remains inspectable). This is the registry's transparency guarantee — equivalent in spirit to `proxy.golang.org` + `sum.golang.org`.

**Install-side** (CI and dev environments):
- **`Cargo.lock` committed and used in CI**. Reproducible builds; tamper-evident (every dep is content-addressed by SHA in the lock).
- **`--locked`** in CI install steps (`cargo build --locked`, `cargo test --locked`). Prevents accidental `Cargo.lock` regeneration during builds.
- **`--frozen`** for fully air-gapped CI variants — disallows network access during dep resolution.
- **No build scripts run as root**: not applicable; Cargo runs build scripts in the user context. The closest concern is `build.rs` files in deps that execute arbitrary code at build time. Cargo has no built-in "ignore-scripts" equivalent, but the `cargo deny` check's `bans` table can blocklist specific crates known to ship malicious `build.rs`. Combine with `cargo geiger` (post-1.0) to audit `unsafe` usage in the dep tree.

**Dependency surface**:
- **`cargo audit`** in `ci.yml`: `cargo install cargo-audit && cargo audit --deny warnings`. Fails CI on any RustSec advisory affecting the resolved dep set.
- **`cargo deny check`** in `ci.yml`: enforces license policy (only MIT/Apache-2.0/BSD permissive), source policy (only crates.io), advisory check (RustSec), and ban list (no `openssl-sys` if we want pure-Rust TLS; no known-bad crates). Configured in `deny.toml`.
- **Dependabot** at `.github/dependabot.yml` — weekly schedule, grouped security updates. Minimum config:
  ```yaml
  version: 2
  updates:
    - package-ecosystem: cargo
      directory: "/"
      schedule: { interval: weekly }
      groups:
        security: { applies-to: security-updates, patterns: ["*"] }
        minor:    { update-types: ["minor", "patch"], patterns: ["*"] }
      open-pull-requests-limit: 5
    - package-ecosystem: github-actions
      directory: "/"
      schedule: { interval: weekly }
      groups:
        actions: { patterns: ["*"] }
  ```
  Groups prevent the dep PR flood (e.g., one PR for all `serde-*` bumps). `open-pull-requests-limit: 5` keeps Dependabot from monopolizing CI.
- **Minimal hard runtime deps** (§5.1): reqwest + tokio + serde stack + thiserror + uuid + httpdate + bytes + futures-core + tracing + http + url + rand. Roughly 10 direct deps; the transitive graph is ~80 crates dominated by tokio + reqwest's hyper stack. Comparable to anthropic-rust / openai-rust.

**Static analysis**:
- **CodeQL does NOT support Rust** as a first-class language at plan time (rust-analyzer + clippy cover the same niche). No CodeQL workflow. Same situation as PHP.
- **`cargo clippy`** with the pedantic group selectively enabled covers most security-adjacent patterns (integer overflows, panics in libraries, unwrap-on-Option, etc.).
- If a future security review identifies a specific concern, add `cargo geiger` (for unsafe-code surveying of the dep tree) or `cargo-careful` (for runtime UB detection in tests). Don't pay the dual-toolchain maintenance cost prophylactically.

**Workflow hardening**:
- **Pin GitHub Actions to commit SHAs**, not floating tags. `uses: actions/checkout@<40-char-sha>  # v4.2.2`. Same for `dtolnay/rust-toolchain`, `Swatinem/rust-cache`, etc.
- **Restrict workflow permissions**. Each job declares only what it needs. `ci.yml` is read-only on contents; `integration.yml` only needs `secrets:` access; no release workflow exists.

**Out of scope for v1.0**:
- SBOM generation (CycloneDX or SPDX). `cargo-cyclonedx` exists and works well; worth adding later, not blocking.
- Sigstore signing of crates. There's no crates.io-side support yet; the [Trusted Publishing pre-RFC](https://github.com/rust-lang/rfcs/pull/3691) addresses this. Revisit when it ships.

## 17. Best practices baked in (Stripe / Anthropic / Tokio playbook)

These are the explicit reasons behind specific decisions above. Keep them in mind as principles, not just rules:

- **Async-first with optional blocking** (anthropic-rust, openai-rust, stripe-rust, reqwest): async is the primary surface; the blocking wrapper is `tokio::runtime::block_on` over the same async code. Single code path, two ergonomic API surfaces.
- **Single `Error` enum with `thiserror`** (anthropic-rust, openai-rust, reqwest): pattern-matching is the idiomatic dispatch; `matches!(err, Error::Auth { .. })` covers the common case; predicate methods kept for spec parity.
- **Auto-retry with idempotency keys** (Stripe / Anthropic): every POST sends a UUID4; callers override via `ProjectModeInput::idempotency_key`. 5xx + 429 + network + timeout retried; 4xx never.
- **`tracing` for observability** (anthropic-rust, openai-rust, aws-sdk-rust, tokio): the ecosystem-standard structured-logging crate. Silent by default; plugs into every observability backend via subscribers.
- **Native sum types** (Rust idiom): `enum RenderInput` is the closest expression of "exactly one of these two shapes" the language allows — cleaner than every sibling SDK's workaround (PHP sealed-abstract-class, Go sealed-marker-interface, Python TypedDict union).
- **Zero `unsafe`** (`#![forbid(unsafe_code)]`): assertable at compile time; signals supply-chain posture.
- **Builder pattern + sensible defaults** (anthropic-rust, openai-rust): `PoliPage::new(api_key)` for the 95% case; `PoliPage::builder()...build()` for advanced configuration. Compile-time-enforced via type-state would be over-engineering at v1.0; the runtime-checked `build()` returning `Result` is sufficient.
- **One-client-per-process pattern** (reqwest convention): cloning `PoliPage` is cheap (`Arc::clone`); the underlying `reqwest::Client` pools connections automatically. Document prominently.
- **First-class cancellation via dropping futures + `tokio::time::timeout`** (Rust async idiom): no `AbortSignal`-equivalent needed; the runtime gives us cancellation for free.
- **Stable wire format**: camelCase on the wire (via `#[serde(rename_all = "camelCase")]`), snake_case in Rust. The derive macro is the translator; users never write conversion code.
- **Type-safe input modes via native enum**: `ProjectModeInput` vs `InlineModeInput` enforced at compile time. No runtime check needed; the type system makes the wrong call un-typeable.
- **`Option<T>` for nullable fields** (Rust idiom): no need for `Ptr<T>` / `Opt[T]` helpers.
- **Idempotency for state-mutating calls**: every POST gets `Idempotency-Key`; deletes are inherently idempotent (200/204 even on already-deleted).
- **No silent fallback to old endpoints**: if a future API change breaks the SDK, fail loudly with a clear error.
- **Predictable versioning**: SemVer 2.0.0; `cargo public-api --diff` enforces in CI. v1.0.0 = behavior parity with `@poli-page/sdk@1.0.0`.
- **Named constants in `internal/constants.rs`**: every API path, every default (`DEFAULT_BASE_URL`, `DEFAULT_MAX_RETRIES`, `DEFAULT_RETRY_DELAY`, `DEFAULT_TIMEOUT`, `RETRY_AFTER_CAP`), every header name. Tests import from there — never re-type string literals.
- **`#![warn(missing_docs)]` + `-D warnings`** (Rust convention): doc comments on every public symbol, enforced in CI. docs.rs becomes the canonical reference automatically.
- **Doctests in every example**: `cargo test --doc` exercises the README examples + every method's `///` example. Examples can't bit-rot.
- **Dual-licensed MIT OR Apache-2.0**: Rust-ecosystem convention; maximizes downstream compatibility.
- **Supply-chain protections** (§16.6): `Cargo.lock` committed + `--locked` in CI + `cargo audit` + `cargo deny` + Dependabot + SHA-pinned actions + per-crate scoped publish token. Equivalent to Node `--provenance`, Python wheels-only + lockfile + pip-audit, Go `go.sum` + `govulncheck`, and PHP `composer audit`. Crates.io Trusted Publishing isn't shipped yet; revisit when it lands.
- **Forward-compat enum + response shape** (stripe-rust, anthropic-sdk-rust): `#[serde(other)]` catch-all on extensible enums; **never** `#[serde(deny_unknown_fields)]` on response types. A server-side field or enum addition is a silent no-op, not a deserialization error — old SDKs keep working.
- **Per-attempt timeout + overall elapsed budget** (aws-sdk-rust): document explicitly that `timeout` is per-attempt; expose `max_elapsed_total` for callers that need a hard SLA. Never let retries silently exceed a caller's expectations.
- **Request-ID on every span** (cloudflare-rs, aws-sdk-rust): `x-request-id` recorded as a `tracing` span field on both success and error paths so log correlation works on the happy path, not just when something breaks.
- **Both `cargo public-api --diff` and `cargo semver-checks`** in CI (tokio, axum, clap): the two catch different classes of semver violations; the cost of running both is ~10 seconds.
- **Feature-gated items annotated for docs.rs** (`#[cfg_attr(docsrs, doc(cfg(...)))]`): users browsing docs.rs see exactly which Cargo features unlock which APIs.

## 18. Divergences from the Python / Go / PHP / Node SDK plans (intentional)

These are places where the Rust SDK diverges from siblings because Rust has a stronger or more standard primitive (or a different ecosystem norm). All other behavior matches exactly. New divergences require explicit discussion — do not add them silently.

| Concern | Node / Python / Go / PHP | Rust | Why |
|---|---|---|---|
| Concurrency | sync+async (Python) / sync+goroutines (Go) / sync (PHP) / async (Node) | **Async-first + optional `blocking` feature** | Matches every leading Rust SDK (anthropic-rust, openai-rust, stripe-rust). One code path, two surfaces via `block_on`. |
| HTTP client | built-in (`fetch` / `httpx` / `net/http`) / PSR-18 injection (PHP) | **`reqwest` as a hard dep** | De facto Rust HTTP client; no PSR-equivalent ecosystem. Custom-client injection deferred (non-breaking to add later). |
| Sum types | TypeScript discriminated unions / Python `Union[]` / Go sealed marker interface / PHP sealed abstract class | **Native `enum RenderInput`** | Rust has real sum types. The cleanest expression across all SDKs. |
| Errors | single class (Node/Go) vs hierarchy (Python/PHP) | **Single `enum Error` with variants** (thiserror) | Pattern-matching is idiomatic Rust; the enum gives hierarchy semantics without inheritance gymnastics. |
| Nullable wire fields | `string \| null` / `Optional[str]` / `*string` / `?string` | **`Option<T>`** | Rust language primitive. No helper needed. |
| Wire translation | hand-written `to_wire`/`from_wire` (Python/PHP) / `json:` field tags (Go) / native (Node) | **`#[serde(rename_all = "camelCase")]` derive** | Zero hand-written translators; the macro is the conversion. |
| Cancellation | `AbortSignal` (Node) / `asyncio.CancelledError` (Python) / `context.Context` (Go) / `timeout:` only (PHP) | **Future-drop + `tokio::time::timeout`** | Rust async runtime gives cancellation for free via Drop. |
| Streaming | `ReadableStream` / generator / `io.ReadCloser` / PSR-7 `StreamInterface` | **`impl Stream<Item = Result<Bytes>>`** + `impl Read` for blocking | Rust-native composability with `futures` ecosystem. |
| Constructor | options object / dict / functional options / named arguments | **Builder pattern (`PoliPage::builder()...build()`) + shortcut `PoliPage::new(api_key)`** | Rust has no named arguments. Builder is the ecosystem-standard alternative. |
| Observability | logging (Python) / log/slog (Go) / PSR-3 (PHP) / hooks (Node) | **`tracing` crate** | Rust ecosystem standard; integrates with every observability backend. |
| Compile-time type checks | TS compiler / pyright / Go compiler / PHPStan | **rustc + clippy + trybuild for "must-fail" cases** | Type errors are build errors in Rust; trybuild adds confidence for new contributors. |
| Doc generation | typedoc / pdoc / pkg.go.dev / phpDocumentor | **rustdoc on docs.rs** | Zero-config; auto-built on publish. |
| Lockfile | committed (Node) / committed (Python lock) / committed (`go.sum`) / NOT committed (PHP) | **`Cargo.lock` committed** | Modern Cargo recommendation (2023+) for libraries too. |
| Install-smoke | npm pack + tarball install / wheel install / consumer module / `composer require` from path | **`cargo new` consumer crate + `path = "..."` dep** | Mirrors the consumer experience. |
| License | MIT (all four) | **MIT OR Apache-2.0 dual** | Rust ecosystem convention; maximizes compatibility. |
| Vuln scanning | `npm audit` / `pip-audit` / `govulncheck` / `composer audit` | **`cargo audit` + `cargo deny check`** | RustSec advisory DB; `deny` adds license + source policy. |
| Static security | CodeQL (Node/Python/Go) / none (PHP) | **None for v1.0** (CodeQL doesn't support Rust) | `clippy` + `cargo audit` + `cargo deny` cover the realistic threat model. Same posture as PHP. |
| Publishing | npm publish (Node) / Trusted Publishing (Python) / git tag → proxy.golang.org (Go) / git tag → Packagist webhook (PHP) | **`cargo publish` from maintainer's machine + manual git tag** | No Trusted Publishing on crates.io yet; per-crate scoped token is the credential boundary. |
| Release workflow | `workflow_dispatch` Trusted Publishing (Python) / none (Go/PHP) | **None — only `scripts/release.sh`** | No CI-side artifact to build; `cargo publish` is local-only. Matches Go/PHP posture. |
| Hosting | docs.poli.page reverse proxy (all four) | **`docs.rs` direct + optional redirect** | docs.rs IS the canonical Rust docs host. |

All design decisions are locked. What remains is operational tasks and phase-time verifications.

### Pre-flight check (Phase 0, ~1 minute)

- **crates.io name availability**: visit `https://crates.io/crates/poli-page` (HTTP 404 = available). If taken, fallback to `polipage` (no hyphen — many Rust crates do this).
- **`poli_page` library name conflict check**: search docs.rs for any existing crate exporting a `poli_page` library identifier; rare but worth a quick check.

### Future enhancements (post-v1.0)

- **`reqwest-middleware` integration**: expose `with_http_client(reqwest::Client)` builder method for users who want custom middleware (retries are ours, but tracing, caching, rate-limiting middleware would compose). Non-breaking to add as a 1.x minor.
- **Framework integration crates**: `poli-page-axum` (auto-derive PDF response types), `poli-page-tower` (tower service wrapper) — separate crates, separate plan, post-1.0.
- **`tower::Service` implementation**: idiomatic Rust trait for "a thing that takes a request and returns a future of response". Would let users compose our client with the tower middleware ecosystem. Worth investigating once the public API is stable.
- **SBOM generation**: `cargo cyclonedx --format json > sbom.json` in CI; attach to GitHub releases. Worth adding when crates.io Trusted Publishing ships.
- **`cargo-vet`**: Mozilla's supply-chain audit tool (used by `aws-sdk-rust`, `cloudflare-rs`). Adds an `audits.toml` declaring which dep versions have been human-reviewed. Higher operational cost than `cargo audit` (RustSec advisories only) but stronger guarantee. Worth adopting once the dep surface stabilizes post-v1.0.
- **`insta` snapshot tests**: golden-snapshot the wire JSON for every request shape (`tests/snapshots/*.snap`). Catches accidental `#[serde(rename = ...)]` regressions that no other test type catches. Used by `aws-sdk-rust`, `ruff`. Cheap to add; high signal — promote into the v1.x line if the wire-translation layer ever sees a bug.
- **`bon` for input builders**: migrate `ProjectModeInput` / `InlineModeInput` / `ThumbnailOptions` from bare structs with `..Default::default()` to `#[derive(bon::Builder)]`. Compile-time enforcement of required fields (forgetting `data` becomes a build error, not a silent empty body). Used by `anthropic-sdk-rust`. Non-breaking to add as a 1.x minor — keep the struct as the public type, add the generated builder alongside.
- **`release-plz` for release automation**: generates the CHANGELOG diff and version bump from conventional commits; still requires a human for `cargo publish` (no crates.io Trusted Publishing yet). Most-recommended release tool in the Rust ecosystem as of 2026. Evaluate as a replacement for hand-rolled `scripts/release.sh` once the manual cadence proves painful.
